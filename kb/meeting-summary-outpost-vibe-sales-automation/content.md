# Meeting Summary: Outpost + Vibe — Sales & Automation Discovery Call

## 🎯 TL;DR (Executive Summary)
Встреча команд Outpost/Jun и PLUS Vibe для обсуждения автоматизации процессов продаж и маркетинга. Ключевые боли — большое трение в воронке продаж, неквалифицированные лиды и высокий процент отвала (особенно у Jun). Основная цель — автоматизировать воронку для коротких букингов (до 3 месяцев), чтобы вдвое сократить штат сейлзов. Приоритетными направлениями для разработки определены автоматическая верификация, скоринг и "умное" распределение лидов, а также создание единой системы дашбордов для менеджмента.

## 🛠 Упомянутые системы (Entities)
- **CRM/Tools**: HubSpot, Nebo (CRM), Klaviyo, Stripe, Yale (verification), Obligo, apartments.com.
- **Метрики**: Response time, lead drop-off rate (отвал), sales performance/conversion, lead score.

## 📌 Ключевые решения и инсайты
- [Инсайт] Основная цель автоматизации — сократить штат сейлз-команды в 2 раза, начав с полной автоматизации коротких букингов (до 3 месяцев).
- [Инсайт] У Jun 9% отвал лидов на последнем шаге из-за сбора полной суммы в конце, в то время как Outpost собирает security-депозит в начале, что значительно снижает отток. Проблему должен решить сервис Obligo.
- [Решение] Внедрить "умное" распределение лидов (smart-дистрибуцию) на основе перформанса сейлзов, их специализации (длинные/короткие сделки) и часовых поясов.
- [Инсайт] Присутствие сейлза в воронке увеличивает конверсию на 25%, что необходимо учитывать при проектировании полностью автоматизированных флоу.
- [Решение] Расширить окно для работы с лидами (follow-up) с текущих 4-5 дней за счёт AI-автоматизации.
- [Инсайт] Главные проблемы с лидами — низкое качество, неотзывчивость и необходимость в ручной верификации.
- [Приоритет] Главными приоритетами для разработки определены: 1. Верификация лидов, 2. Автобукинг, 3. Скоринг и перераспределение нагрузки на сейлзов.

#### Автоматизация — видение и ограничения
- **Мечта:** полностью автоматический sales process без вовлечения человека
- **Короткие букинги (до месяца)** — реально автоматизировать (аналогия с Airbnb)
- **Длинные букинги (6–12 мес.)** — нужны доверие, живая коммуникация, часто офлайн-показы
- 50% сделок Outpost закрываются без показов (фото, 3D-туры, раппорт с сейлзом)
- У Jun 42% букингов — до 3 месяцев → первые кандидаты на автоматизацию
- **Цель:** сократить штат сейлзов в 2 раза

#### Три ключевые проблемы с лидами
- **Неквалифицированные / lacking detail leads** — после слияния постят листинги везде → больше мусорных лидов; нужен AI для сбора данных и предквалификации
- **Non-responsive leads** — сейчас окно follow-up 4–5 дней, потом лид убивается; с AI окно можно расширить
- **Верификация** — автоматический сбор данных и проверка до передачи сейлзу

#### Скоринг и дистрибуция лидов
- **Скоринг по дате заезда:** заезд через 7–8 мес. → конверсия минимальная → на автоматику (Jack)
- Учёт часовых поясов сейлзов при назначении лидов
- **Smart-дистрибуция по перформансу:** больше лидов тем, кто больше закрывает (Alex Kos)
- Двухнедельный цикл пересчёта, чтобы сейлзы «всегда бежали»
- Учёт навыка закрытия длинных vs коротких сделок (Mike)
- **Сейчас системы разные:** Outpost — 3 блока по продолжительности + ручной пересчёт; Jun — equal distribution по шифтам

#### Дашборды и метрики
- Real-time дашборды для менеджеров 3 команд (Mike)
- Response time, активность сейлзов, подбор лидов
- Система должна сигнализировать, если сейлз получает лидов, но не отвечает
- В Outpost все на 1099, чисто на сделке → важно трекать работу на шифтах

#### Outpost vs Jun — различия в воронке
- **Outpost:** security-депозит в начале → финальный invoice = половина суммы → после подписания LIS клиент не соскакивает
- Основной tension: между approve и подписанием LIS
- **Jun:** не собирает ничего до финала → крупный финальный invoice (7k+) → шок → 9% отвал
- Лиды 50/50 сайт / платформы (apartments.com и др.)
- Клиент может зарегистрироваться без выбора конкретной комнаты (по бюджету)
- **Конверсия с сейлзом на 25% выше**, чем без
- Obligo подключается 1 марта для решения проблемы с депозитами

#### Коммуникации
- **Каналы:** SMS, email, звонки
- **Системы:** HubSpot (Outpost), Klaviyo (маркетинговые flow), CRM Nebo
- **Проблема:** коммуникации разрозненные, нужно склеить в единую ленту
- **Ключевая точка роста** — все коммуникации в одном месте для smart-решений

#### Sales Workflow (Outpost) — пошагово
1.  **Лид попадает в систему** — клиент на сайте выбирает дом/район, даты (мин. 30 дней), видит цену, нажимает Book
2.  **Автоматический acknowledge** — HubSpot отправляет email, данные → CRM Nebo
3.  **Назначение сейлза** — дистрибуция по текущим правилам
4.  **Sales-driven коммуникация** — сейлз проверяет availability (на сайте не показывается), формирует offer, предлагает альтернативы; коммуникация: звонки, SMS, email
5.  **Клиент соглашается** — сейлз создаёт букинг в Nebo → «Start Sales Flow»
6.  **Онбординг-воронка** (автоматическая, по линку, без личного кабинета):
    -   Application → оплата invoice → security-депозит
    -   Если клиент стопорится — SMS/email: «24 часа до отмены»
7.  **Параллельная верификация:**
    -   Stripe: ID verification, фото, история диспутов
    -   Кредитное бюро (если SSN): тенант-чек, криминал рекорды, court records
    -   12+ мес.: доп. документы, ручная проверка подлинности (Yale)
    -   Студенты: отдельный flow для гарантора ($20 fee + подписание)
8.  **Approve** → автоматическое уведомление клиенту
9.  **Post-approve:** подписание контракта (LIS) → invoice за первый месяц → protection plan
10. **Move-in** — за 7 дней до заезда credentials для личного кабинета → онбординг → move-in task → передача в ops (Тина)

**Jun (отличия):** лиды 50/50 сайт/платформы; клиент может начать без выбора комнаты (по income/бюджету); догонялки через Klaviyo; security-депозит не собирается → проблема с collection на финале.
